# ADR 0005: API 版本化与统一响应 Envelope

## 状态

已采纳

## 背景

项目需要以 `/api/v1/*` 作为统一稳定入口，避免控制台、CLI、MCP、HTTP capability 四种入口继续混用裸响应和未版本化路径，导致后续演进持续漂移。

## 决策

- 稳定 HTTP 基线路径统一为 `/api/v1/*`。
- 不再保留旧 `/api/*` 的兼容入口，所有接入方统一切到 `/api/v1/*`。
- 普通 JSON 成功响应统一包装为 `{ data, meta, traceId, error }`。
- SSE、重定向和特殊流式协议不强制套 envelope。

## 取舍

- 优点：接入方契约稳定，traceId/requestId 可以统一透传，文档与能力平台更容易对齐。
- 成本：前端、CLI、文档和示例必须同步迁移到 `/api/v1/*` 与 envelope 契约。

## 后续影响

- 新增接口必须优先声明 `/api/v1/*`。
- 文档、示例、能力注册表和测试都要以 `/api/v1/*` 为唯一基线。
